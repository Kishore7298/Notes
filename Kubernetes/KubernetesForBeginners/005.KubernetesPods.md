## ğŸ“¦ What is a Pod?
- A Pod is the smallest unit in Kubernetes
- It encapsulates one or more containers
- Pods are used instead of directly deploying containers to worker nodes
- Commonly, each pod contains only one main container

## ğŸ”„ Scaling Applications
To scale, you create more pods, not more containers inside the same pod.

### Example:
- One pod = One container (web app)
- To scale to handle more users: create multiple identical pods

## ğŸ§© Multi-Container Pods
Rare but valid use cases when containers need to work tightly together:
- Main web app + helper container
- Sidecar patterns

### Features in Multi-Container Pod:
| Feature | Description |
|---------|:------------|
| Networking | Share network namespace (communicate via localhost) |
| Storage | Can share volumes between containers |
| Lifecycle | Start and stop together |

## ğŸ³ Comparison to Docker Workflow

### Without Kubernetes:
- Use `docker run` to start containers
- Need to manually:
  - Link containers
  - Share volumes
  - Monitor lifecycles
  - Scale by running multiple commands

### With Kubernetes:
- Pods automate:
  - Networking
  - Storage
  - Lifecycle syncing
- Easier to manage complex apps with helper containers

## ğŸ› ï¸ How to Deploy a Pod

### Create a Pod:
```bash
kubectl run nginx --image=nginx
```
This creates a pod running an Nginx container.

> Note: Kubernetes pulls the image from Docker Hub by default (unless configured otherwise)

### View Pod Status:
```bash
kubectl get pods
```
Status progression:
1. ContainerCreating
2. Running

## ğŸŒ Accessing the Pod
- Pod is not accessible externally by default

## ğŸ§  Key Takeaways
- A pod usually wraps one container (simple apps)
